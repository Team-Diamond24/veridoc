System Overview

What VERIDOC Does

VERIDOC is a four-layer AI system that evaluates government tender bids. It reads eligibility rules from the tender document, parses each bidder's submission (in any format), extracts evidence, and produces explainable pass/fail/review verdicts with full audit trails.

Core Principle

Never silently reject. If the system is unsure, it flags the case for human review with clear reasoning. This ensures fairness and legal defensibility in government procurement.

The Big Picture

Tender Document Bidder Submissions ↓ ↓ ┌───────────────────────────────────────┐ │ LAYER 1: Document Ingestion & OCR │ │ Convert all docs → structured text │ └──────────────────┬──────────────────┘ ↓ ┌───────────────────────────────────────┐ │ LAYER 2: Obligation Extraction │ │ Parse tender → structured rules │ └──────────────────┬──────────────────┘ ↓ ┌───────────────────────────────────────┐ │ LAYER 3: Evidence Mapping │ │ Find proof → chain evidence │ └──────────────────┬──────────────────┘ ↓ ┌───────────────────────────────────────┐ │ LAYER 4: Verdict Engine │ │ Evaluate evidence → PASS/FAIL/REVIEW │ └──────────────────┬──────────────────┘ ↓ Evaluation Report (with PDF highlights, audit trail, counterfactual guidance)
The Four Layers Explained

Layer 1: Document Ingestion & OCR

Purpose

Convert every document into machine-readable text with precise location metadata. This is the foundation — garbage in means garbage out.

What It Handles

  • Typed PDFs → Extract text + structure directly
  • Scanned PDFs → OCR with confidence scoring
  • Word documents (.docx) → Parse with python-docx
  • Excel files (.xlsx) → Extract cells as structured data
  • Images (JPG, PNG, etc.) → OCR with orientation detection
  • Mixed documents → Handle page-by-page (some typed, some scanned)

Key Feature: Bounding Boxes

Every extracted piece of text carries metadata: { text: "₹50 crore", page: 5, bbox: [100, 200, 300, 220], confidence: 0.94 }

This lets us later highlight the exact PDF location when showing the officer our reasoning.

Technologies

PaddleOCR-VL 109 languages, Hindi/regional scripts
python-docx Word document parsing
openpyxl Excel parsing
PDF.js PDF structure analysis

Output Format

{ "document_id": "bid_001_company_abc", "page_count": 42, "extracted_text": "...(markdown)...", "chunks": [ { "text": "Annual turnover ₹50 crore", "page": 5, "bbox": [100, 200, 300, 220], "ocr_confidence": 0.94, "source": "P&L statement", "chunk_id": "chunk_042" } ] }

Layer 2: Obligation Extraction

Purpose

Extract the eligibility rules from the tender document into structured, machine-readable form. Understand what the tender actually requires.

What It Extracts

  • Criterion name (e.g., "Annual Turnover")
  • Modal type (MANDATORY, CONDITIONAL, OPTIONAL)
  • Threshold (e.g., "₹5 crore minimum")
  • Temporal scope (e.g., "last 5 years")
  • Evidence type (e.g., "audited P&L statement")
  • Dependencies (e.g., "Criterion A only applies if Criterion B is met")

The Obligation Graph

Criteria are modeled as a directed acyclic graph (DAG). If Criterion A fails, any criteria that depend on it are automatically marked as "blocked."

Criterion 1: GST Registration [MANDATORY] ↓ Criterion 2: GST Certificate [MANDATORY, depends on 1] ↓ Criterion 3: GST History (3+ years) [MANDATORY, depends on 2]

Technologies

LegalBERT Fine-tuned on CRPF tenders (F1 ≥ 0.75)
Deontic Logic Parse modal language (must, should, may)
Spacy NLP Dependency parsing for relationships

Output Format

{ "criterion_id": "crit_001", "name": "Annual Turnover", "modal": "MANDATORY", "threshold": { "value": 5, "unit": "crore", "currency": "INR" }, "temporal": { "period": 5, "unit": "years" }, "evidence_type": "audited_financial_statement", "dependencies": [], "page_references": [3, 4] }

Critical Validation Gate

Phase 1, Week 2: Test LegalBERT on hold-out CRPF tenders. Measure F1 on obligation extraction (modal type, threshold, evidence type). If F1 < 0.75, expand training data before proceeding.

Layer 3: Evidence Mapping

Purpose

For each criterion, find and chain together all relevant evidence from the bidder's documents. Don't just find individual documents — build complete proof chains.

The Problem with Naive Retrieval

If you ask "Does the bidder have ₹5 crore turnover?" and retrieve one chunk that says "We have annual turnover ₹50 crores," you're missing the chain:

1. Bid letter claims: "Annual turnover ₹50 crores" 2. Supported by: P&L statement (pages 4-5) 3. Certified by: Auditor certificate ("P&L verified for FY2021-23") 4. Verified by: CA registration + firm details

Naive RAG would retrieve just the claim. IndexRAG retrieves the entire chain in one pass.

How IndexRAG Works

  • Indexing phase: For each document chunk, pre-compute "bridges" to related chunks (same topic, same time period, same entity)
  • Retrieval phase: Given a criterion requirement, retrieve not just matching chunks but the full chain of connected chunks
  • Scoring: Rate chain completeness ("Do we have all the evidence needed?") and confidence

External Evidence Flagging

Some evidence isn't in the documents — it's external (requires manual lookup). We explicitly flag these:

Criterion Type Evidence Location VERIDOC Action
Internal (e.g., turnover) In bid documents Retrieve full chain
External (e.g., no debarment) Government registry Generate checklist for officer
Certification (e.g., ISO) Certificate in bid + auditor registry Retrieve cert, suggest manual verification

Technologies

IndexRAG Multi-hop evidence chaining
Faiss/Milvus Vector similarity search
IndicBERT Embeddings for Indian names/text

Output Format

{ "criterion_id": "crit_001", "evidence_chain": [ { "step": 1, "text": "Annual turnover ₹50 crores", "source": "Bid letter, page 2", "chunk_id": "chunk_042", "confidence": 0.96 }, { "step": 2, "text": "[P&L statement showing revenue ₹50 Cr]", "source": "Financial statements, pages 4-5", "chunk_id": "chunk_156", "confidence": 0.92 }, { "step": 3, "text": "P&L verified by Chartered Accountant", "source": "Audit certificate, page 6", "chunk_id": "chunk_201", "confidence": 0.94 } ], "chain_completeness": 0.95, "chain_confidence": 0.94 }

Validation Gate

Phase 2, Week 6: Test evidence mapper on 20 real bidder submissions. Measure chain completeness (do we get 2-4 connected chunks per criterion?) and accuracy (do officer reviewers agree with the chains we built?).

Layer 4: Verdict Engine

Purpose

Evaluate the evidence against each criterion and produce a verdict: PASS, FAIL, or REVIEW.

The Three-Axis Framework

Instead of one overall score, we evaluate on three independent axes. This gives clarity on why we made the decision.

Axis What It Measures Example
Axis 1: Evidence Quality Is the evidence legible, complete, and properly formatted? (0–100%) OCR confidence 0.94, all pages present, clear structure
Axis 2: Semantic Match Does the evidence address the criterion? (0–100%) P&L statement matches "financial statement" requirement
Axis 3: Threshold Compliance Does the value meet the requirement? (0–100%) ₹50 crore > ₹5 crore required

Decision Logic

Not a formula, but a decision tree. We explicitly define thresholds (not "weighted products"):

if any(axis1 < 20, axis2 < 20, axis3 < 20): verdict = FAIL # Veto: critical axis is below threshold elif all(axis1 ≥ 80, axis2 ≥ 80, axis3 ≥ 80): verdict = PASS # Strong evidence on all fronts elif any(50 <= axis < 80) or mean(axis1,2,3) in [55, 79]: verdict = REVIEW # Borderline: human judgment needed elif any(axis1 in [20,50]) or any(axis2 in [20,50]): verdict = REVIEW # Moderate evidence on some axes else: verdict = FAIL # Low confidence overall

Per-Axis Thresholds Explained

Axis 1: Evidence Quality

<20%: Unreadable (OCR <0.7, missing pages)

20–50%: Poor quality (OCR 0.7–0.85, partial docs)

50–80%: Fair (OCR 0.85–0.92, minor gaps)

≥80%: High quality (OCR >0.92, complete)

Axis 2: Semantic Match

<20%: Wrong document type (résumé for "financial statement")

20–50%: Partial match (summary instead of detail)

50–80%: Good match (slight format variation)

≥80%: Perfect match (right doc, right format)

Technologies

Claude Sonnet 4 LLM reasoning + structured output
Structured Output Format JSON verdict schema
Rule Engine Decision tree implementation

Output Format

{ "criterion_id": "crit_001", "bidder_id": "bid_001", "axis_scores": { "evidence_quality": 94, "semantic_match": 88, "threshold_compliance": 100 }, "verdict": "PASS", "reasoning": "All three axes strong. P&L clearly shows ₹50Cr > ₹5Cr required.", "pdf_highlights": [ { "page": 5, "bbox": [100, 200, 300, 220] } ], "counterfactual": null, "confidence": 0.96 }

Counterfactual Guidance (FAIL Cases Only)

For every FAIL verdict, we generate: "If you submit [X] covering period [Y] attested by [Z], this criterion passes."

Requirement: GFR 2017 Rule 204 (right to recourse)

Validation Gates

Phase 2, Week 8: Test verdict logic on 20 sample cases with CRPF officers. Validate thresholds are sensible.

Phase 3, Week 12: Validate collusion detection on 50+ real tenders (precision ≥ 85%).

Database Design

PostgreSQL is used for transactional data (documents, verdicts, corrections). Vector embeddings are stored separately in Faiss for fast retrieval.

Core Tables

Table Purpose Key Fields
tenders One row per tender tender_id, name, criteria_count, created_at
criteria Extracted rules from tender criterion_id, tender_id, name, modal, threshold, dependencies
bidders Companies bidding on tenders bidder_id, company_name, registration_number, tender_id
documents Uploaded bid documents doc_id, bidder_id, filename, page_count, file_type, ocr_confidence
chunks OCR-extracted text segments chunk_id, doc_id, text, page, bbox, confidence, embedding_id
verdicts Final decisions per criterion verdict_id, criterion_id, bidder_id, axis_scores, verdict, confidence, counterfactual
corrections Officer feedback for retraining correction_id, verdict_id, axis_wrong, correct_label, officer_id, timestamp, validated
audit_log Complete decision trail log_id, action, user_id, timestamp, details

Relationships (Simplified ERD)

tenders (1) ──→ (many) criteria ├──→ (many) bidders └──→ (many) documents bidders (1) ──→ (many) documents └──→ (many) verdicts documents (1) ──→ (many) chunks ├──→ (many) verdicts └──→ (many) evidence_chains verdicts (1) ──→ (many) corrections └──→ (many) audit_logs
API Endpoints (FastAPI)

The backend exposes REST endpoints for document upload, evaluation, and feedback. All endpoints return JSON.

Tender Management

Endpoint Method Purpose
/tender/upload POST Upload tender PDF → extract criteria → save to DB
/tender/{id}/criteria GET Get extracted criteria for a tender (for review)
/tender/{id}/evaluate POST Start evaluation (takes ~5-10 min)

Bid Evaluation

Endpoint Method Purpose
/bidder/upload POST Upload bid documents (multi-format)
/bidder/{id}/verdicts GET Get all verdicts for a bidder
/bidder/{id}/evidence/{criterion_id} GET Get evidence chain for a specific criterion
/verdicts/summary GET Summary: counts of PASS/FAIL/REVIEW for all bidders

Officer Actions

Endpoint Method Purpose
/verdict/{id}/override POST Officer disagrees → submit correction
/report/export GET Generate final PDF report (tender + all verdicts + audit trail)
/audit-log GET Complete audit trail (all decisions + overrides)

Example Request/Response

POST /verdict/v123/override { "correct_verdict": "PASS", "axis_wrong": 2, "reason": "Evidence clearly matches criterion", "officer_id": "officer_456" } Response: { "correction_id": "corr_789", "status": "captured", "message": "Override recorded. Thank you for the feedback." }
Technology Stack & Justification

Why Each Choice?

PaddleOCR-VL-0.9B

Why: State-of-the-art accuracy on scanned documents + 109 languages including Indian scripts. Self-hosted (no API calls = faster + private).

Cost: $0.09/1K pages self-hosted

LegalBERT

Why: Trained on legal text. Fine-tuning on CRPF tenders teaches it domain language (GFR, PSU, etc.).

Trade-off: Slower than RoBERTa but better at legal reasoning

IndexRAG

Why: Handles complex evidence requiring multiple documents. Naive RAG would miss connections.

Cost: Build time is higher; retrieval is slower (mitigated by smaller search space)

Claude Sonnet 4

Why: Best-in-class reasoning on complex criteria. Structured output mode ensures consistent JSON.

Cost: ~$0.15–0.25 per criterion. Manageable at 500 criteria × 10 bidders = ~$1K per tender

FastAPI

Why: Async Python. Fast, modern, easy to test. Fits well with NLP pipelines.

Alternative: Could use Django REST but FastAPI is lighter

PostgreSQL

Why: Relational + proven. Audit logs need ACID transactions. JSON columns for flexible metadata.

Alternative: Could use MongoDB but PostgreSQL is more audit-friendly

React + PDF.js

Why: Client-side PDF rendering. Allows bounding-box overlays directly on PDF without re-rendering.

Alternative: Could use Svelte but React is more standard

Faiss/Milvus

Why: Fast vector search for evidence retrieval. Self-hosted = data privacy. Faiss for dev, Milvus for production scale.

Cost: Open source

Risks & Mitigations

Critical Risks

Risk 1: LegalBERT Doesn't Learn CRPF Domain

Severity: CRITICAL | Timeline impact: +2–4 weeks

What could go wrong: LegalBERT fine-tuned on 40–50 documents isn't enough to learn CRPF's specific language (abbreviations, modal conventions, regulatory references).

Mitigation: Validation gate at Week 2. If F1 < 0.75 on hold-out tenders, immediately expand to 100+ documents before proceeding to Phase 2.

Contingency: Have a domain expert manually annotate 100 tenders if time permits.

Risk 2: Evidence Mapper Returns Incomplete Chains

Severity: CRITICAL | Timeline impact: +1–2 weeks

What could go wrong: IndexRAG retrieves 10+ disconnected chunks instead of a coherent 2–4 chunk chain.

Mitigation: Phase 2, Week 6 validation. Test on 20 real scenarios. If chains are poor, add post-retrieval consolidation (merge overlapping chunks, drop noise).

Contingency: Fall back to simpler RAG + manual chain assembly for Phase 2.

Risk 3: Officers Don't Trust the System

Severity: HIGH | Timeline impact: +1–2 weeks (more testing needed)

What could go wrong: Officers see verdicts and assume the system is wrong, overriding most decisions rather than using it.

Mitigation: Start in "advisory" mode (first 2 tenders). Show officers the PDF evidence + reasoning side-by-side so they can audit our logic. Iterate UI based on feedback.

Contingency: If adoption is low, consider simplifying verdicts to just "clearly eligible" vs. "clearly ineligible" vs. "ambiguous" (less nuance, but easier to trust).

High-Priority Risks

Risk 4: Collusion Detection False Positives

Severity: HIGH | Timeline impact: +1 week

What could go wrong: OECD data says 81–95% accuracy, but CRPF tenders have different patterns. Identical bids might be legal (template use).

Mitigation: Phase 3, Week 12 validation on 50+ real CRPF tenders. Only show flags with precision ≥ 85%. Mark module as "advisory."

Contingency: If precision is <85%, disable collusion flags for now; label as "future enhancement."

Risk 5: System Drifts with Officer Bias During Retraining

Severity: MEDIUM | Timeline impact: None (caught in monitoring)

What could go wrong: A biased officer consistently overrides verdicts in favor of certain bidder types. Retraining learns the bias.

Mitigation: Implement correction validation gate (Phase 4, Week 16). Senior analyst reviews 10 out of every 50 corrections. Bias dashboard tracks per-officer and per-bidder override rates.

Contingency: If bias is detected (2x higher override rate), flag officer for coaching or audit.

Medium-Priority Risks

Risk 6: Scanned Documents Have Low OCR Confidence

Severity: MEDIUM | Timeline impact: None (expected)

What could go wrong: Some CRPF bids are photocopies of handwritten forms. OCR confidence drops below 0.7.

Mitigation: By design: these get flagged as REVIEW (low Axis 1 score). Officer reviews manually.

Contingency: Have a fallback to manual data entry form for low-confidence docs.

Risk 7: No Coverage for External Evidence (GST registry, debarment DB)

Severity: MEDIUM | Timeline impact: None (by design)

What could go wrong: Some criteria require verification against external databases (is the GST number real? is the bidder debarred?)

Mitigation: Flag as "external evidence" at extraction time. Generate officer checklist: "Verify against [registry] using [link]."

Contingency: Phase 5 (future): integrate with CRPF registry APIs.
